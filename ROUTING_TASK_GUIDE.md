# Гайд по экзаменационной задаче на таблицы маршрутизации

Дата фиксации: 2026-06-17.

Этот файл фиксирует реальный формат задачи с экзамена и гайд друга. Его цель - не дать потерять контекст перед переделкой тренажера.

## Суть задачи

На экзамене задача оказалась не такой, как старая задача в `subnet_task.js`.

Старая задача в основном тренирует:

- подобрать маски;
- разбить пул адресов;
- назначить IP;
- прописать маршруты.

Новая экзаменационная задача тренирует другое:

- дана готовая схема Intranet;
- на схеме уже подписаны IP-адреса интерфейсов и диапазоны адресов LAN;
- стрелкой показан выход в глобальную сеть;
- нужно привести таблицы маршрутизации для каждого маршрутизатора отдельно;
- нужно описать активное и пассивное оборудование, если каждое облако - сегмент коммутируемой сети `1000Base-T`.

Формулировка условия:

> На рисунке показана структура локальной сети Intranet. Стрелкой показан маршрут, ведущий в глобальную сеть, подписаны адреса узлов. Приведите содержимое таблиц маршрутизации всех показанных на схеме маршрутизаторов и опишите активное и пассивное оборудование, если каждое облако - это сегмент коммутируемой сети 1000Base-T.

## Общий вид вариантов

Факты от пользователя:

- задачи на экзамене однотипные;
- меняются соединения между маршрутизаторами;
- количество маршрутизаторов в основном 3 или 4;
- количество LAN в основном 3 или 4;
- нужна не одна общая таблица маршрутизации, а отдельная таблица для каждого маршрутизатора;
- нужно учитывать все сети, включая сети между маршрутизаторами.

Вывод для тренажера:

- старую VLSM-задачу нельзя считать достаточной;
- нужен новый режим задачи на построение routing tables по готовой схеме;
- генератор должен уметь строить топологии из 3-4 маршрутизаторов и 3-4 LAN;
- ответ должен проверяться по каждой таблице маршрутизации отдельно.

## Гайд по маскам

Маска `/N` означает, что первые `N` бит отданы под адрес сети, остальные биты - под адреса узлов.

Пример `/27`:

```text
11111111.11111111.11111111.11100000
255.255.255.224
```

Последний октет считается так:

```text
128 + 64 + 32 = 224
```

Количество адресов в подсети:

```text
2^(32 - prefix)
```

Для `/27`:

```text
32 - 27 = 5
2^5 = 32 адреса всего
32 - 2 = 30 usable-адресов
```

Первый адрес подсети - адрес сети, последний - broadcast. Они не назначаются устройствам.

Пример блоков для `/27` в `10.4.4.0/24`:

```text
10.4.4.0-10.4.4.31      сеть 10.4.4.0/27
10.4.4.32-10.4.4.63     сеть 10.4.4.32/27
10.4.4.64-10.4.4.95     сеть 10.4.4.64/27
...
```

Для `LAN1` из примера:

- диапазон узлов: `10.4.4.38-10.4.4.60`;
- интерфейс R2: `10.4.4.37`;
- минимум нужно покрыть адреса `.37-.60`;
- `/28` дает только 16 адресов всего, этого мало;
- `/27` дает 32 адреса всего и покрывает диапазон;
- сеть: `10.4.4.32/27`;
- broadcast: `10.4.4.63`;
- usable: `10.4.4.33-10.4.4.62`.

Для связей маршрутизатор-маршрутизатор в гайде используется `/30`:

- 4 адреса всего;
- 2 usable-адреса;
- 1 адрес сети;
- 1 broadcast.

Пример:

```text
R1-R2: 10.0.3.17 и 10.0.3.18
Блок /30: 10.0.3.16-10.0.3.19
Сеть: 10.0.3.16/30
Broadcast: 10.0.3.19
```

## Правила заполнения таблиц маршрутизации

1. В таблице каждого маршрутизатора должны быть указаны все сети из схемы.
2. Сначала пишутся `C` / `Connected` - сети, к которым маршрутизатор подключен напрямую.
3. Потом пишутся `S` / `Static` - сети, которые существуют в системе, но не подключены к этому маршрутизатору напрямую.
4. Для `Connected` в next-hop пишется `directly connected` и можно указать, к какому соседу или LAN ведет интерфейс.
5. Для `Static` в next-hop пишется IP соседнего маршрутизатора, через который нужно идти к сети.
6. Next-hop должен быть адресом соседнего маршрутизатора в directly connected сети, а не собственным адресом текущего маршрутизатора.
7. Сеть `0.0.0.0/0` - маршрут по умолчанию.
8. У R1 маршрут по умолчанию идет в глобальную сеть через внешний next-hop.
9. У остальных маршрутизаторов маршрут по умолчанию обычно ведет в сторону R1.

Важно: если до сети есть несколько физически возможных путей, статический маршрут может иметь несколько допустимых next-hop. Для тренажера нужно выбрать каноническое правило, например кратчайший путь до сети или путь, показанный в эталоне.

## Активное и пассивное оборудование для 1000Base-T

Если каждое облако - сегмент коммутируемой сети `1000Base-T`, то внутри облака подразумевается Ethernet-сегмент на базе коммутатора.

Активное оборудование:

- маршрутизаторы `R1`, `R2`, `R3`, `R4`;
- коммутаторы в сегментах `LAN1`, `LAN2`, `LAN3`;
- конечные узлы LAN: ПК, серверы или другие хосты с сетевыми адаптерами;
- внешний шлюз/оборудование глобальной сети со стороны стрелки, если его нужно описывать.

Пассивное оборудование:

- витая пара категории не ниже Cat5e для `1000Base-T`;
- патч-корды;
- розетки RJ-45;
- коннекторы RJ-45;
- патч-панели;
- кабельные каналы, стойки, кроссовое оборудование.

## Типичные ошибки

- Не включить сети между маршрутизаторами в таблицы.
- Написать только LAN-сети, забыв transit-сети `/30`.
- Перепутать connected route и static route.
- В static route указать собственный интерфейс вместо IP соседнего маршрутизатора.
- Перепутать gateway хоста и next-hop маршрутизатора.
- Не добавить маршрут по умолчанию `0.0.0.0/0`.
- Указать для LAN слишком маленькую маску, которая не покрывает весь диапазон адресов.
- Считать количество usable-адресов без адреса интерфейса маршрутизатора в LAN.
- Смешать одну общую таблицу вместо отдельных таблиц для каждого маршрутизатора.

## План генератора v2

Цель генератора: создавать однотипные экзаменационные задачи по готовой схеме Intranet, где пользователь сам вычисляет сети и заполняет таблицы маршрутизации каждого маршрутизатора.

Генератор не должен быть похож на старую VLSM-задачу. Здесь не нужно просить пользователя распределять общий пул с нуля. Нужно дать готовую схему с подписанными IP-адресами и диапазонами LAN, а затем проверить выводы пользователя.

### Что должен генерировать

1. Топологию:
   - 3 или 4 маршрутизатора;
   - 3 или 4 LAN-сегмента;
   - один маршрутизатор с выходом в глобальную сеть;
   - router-to-router связи между маршрутизаторами;
   - LAN-сегменты, подключенные к разным маршрутизаторам.

2. IP-адреса:
   - для каждой router-to-router связи два usable-адреса из `/30`;
   - для глобального линка тоже `/30`;
   - для каждой LAN:
     - IP интерфейса маршрутизатора;
     - диапазон адресов узлов, например `10.4.4.38-10.4.4.60`;
     - диапазон должен полностью помещаться в минимальную подходящую подсеть.

3. Условие:
   - текст как в экзамене;
   - схема с роутерами, облаками LAN, IP интерфейсов и диапазонами узлов;
   - стрелка в глобальную сеть.

4. Эталон:
   - список всех сетей;
   - таблица маршрутизации для каждого маршрутизатора;
   - описание активного/пассивного оборудования.

### Топологии

Для первой версии генератора достаточно заранее описанных шаблонов, а не полностью случайного графа.

Подходящие шаблоны:

- 3 роутера цепочкой: `R1 - R2 - R3`, LAN распределены по всем или по двум роутерам;
- 3 роутера треугольником: `R1-R2`, `R2-R3`, `R1-R3`;
- 4 роутера цепочкой: `R1 - R2 - R3 - R4`;
- 4 роутера с развилкой от верхнего роутера, как в текущем примере;
- 4 роутера с одним дополнительным резервным линком, если нужно тренировать выбор кратчайшего next-hop.

На первом этапе лучше использовать только шаблоны без равнозначных путей. Так меньше спорных ответов.

### Генерация адресов

Router-to-router сети:

- всегда `/30`;
- адрес сети должен быть кратен 4 в последнем октете;
- usable-адреса: `network + 1` и `network + 2`;
- broadcast: `network + 3`;
- на схеме подписываются только usable-адреса интерфейсов.

LAN-сети:

- выбирается префикс из `/29`, `/28`, `/27`, `/26`;
- внутри LAN выбирается IP интерфейса маршрутизатора;
- затем выбирается диапазон хостов так, чтобы:
  - router IP и весь host range лежали внутри одной подсети;
  - host range не включал адрес сети и broadcast;
  - по диапазону нельзя было взять меньшую маску;
  - пример: если адреса покрываются `/27`, то `/28` должен быть недостаточен.

Глобальная сеть:

- тоже `/30`;
- IP глобального next-hop и IP интерфейса внешнего роутера должны быть двумя usable-адресами;
- для маршрута по умолчанию на root-router используется внешний next-hop.

### Построение эталона

Эталон строится алгоритмом, не ручным списком.

1. Для каждого сегмента вычислить:
   - `network`;
   - `prefix`;
   - `mask`;
   - `broadcast`;
   - usable-диапазон.

2. Для каждого маршрутизатора найти connected-сети:
   - все сегменты, где этот маршрутизатор является endpoint;
   - тип строки `C`;
   - next-hop/interface: `directly connected`.

3. Для каждого маршрутизатора найти static-сети:
   - все сети схемы, которые не являются connected для этого маршрутизатора;
   - тип строки `S`;
   - next-hop = IP соседнего маршрутизатора на первом линке кратчайшего пути.

4. Добавить default route:
   - на root-router: `S* 0.0.0.0/0 via externalNextHop`;
   - на остальных: `S* 0.0.0.0/0 via next-hop в сторону root-router`.

5. Если есть несколько кратчайших путей:
   - либо такие топологии пока не генерировать;
   - либо явно принять каноническое правило: выбирать соседа с меньшим номером роутера;
   - в UI желательно указать, что ожидается один канонический вариант.

### Проверка ответа пользователя

Проверка сетей:

- сеть может быть введена как `10.0.3.16` + `/30` или как `10.0.3.16/30`;
- маску принимать и как `/30`, и как `255.255.255.252`;
- адрес сети должен быть именно network address, а не IP интерфейса;
- prefix должен совпадать с эталоном.

Проверка таблиц:

- порядок строк в таблице не должен быть важен;
- проверять строки как множество;
- для `C` проверять тип, сеть, маску и `directly connected`;
- для `S` и `S*` проверять тип, сеть, маску и next-hop;
- next-hop должен быть IP соседнего маршрутизатора, не собственный IP;
- все сети схемы должны присутствовать в каждой таблице;
- default route должен быть отдельной строкой `0.0.0.0/0`.

Проверка оборудования:

- активное: маршрутизаторы, коммутаторы, конечные узлы/сетевые адаптеры;
- пассивное: витая пара Cat5e/Cat6, RJ-45, патч-корды, патч-панели, кабельная инфраструктура;
- проверка должна быть мягкой по формулировкам, потому что это текстовый ответ.

### Автотесты для генератора

Нужен отдельный тестовый прогон без браузера.

Минимум на 1000-10000 сгенерированных задач:

- все IP валидные;
- все `/30` router-to-router сети не пересекаются;
- LAN-сети не пересекаются с router-to-router и между собой;
- каждый LAN range полностью лежит в своей сети;
- router IP в LAN лежит в той же сети;
- ни один endpoint IP не равен network/broadcast;
- для каждой таблицы есть все сети схемы;
- connected-сети совпадают с физическими подключениями роутера;
- static next-hop всегда является адресом соседнего роутера в directly connected сети;
- default route у всех роутеров ведет в сторону глобальной сети;
- эталонный ответ, поданный в валидатор, дает полный балл.

Дополнительно полезны негативные тесты:

- собственный IP вместо next-hop должен считаться ошибкой;
- IP интерфейса вместо адреса сети должен считаться ошибкой;
- пропущенная transit-сеть `/30` должна снижать балл;
- неверный prefix при правильном network должен считаться ошибкой;
- отсутствие default route должно считаться ошибкой.

### Порядок внедрения

1. Вынести текущую фиксированную задачу v2 в структуру `template + generatedTask + expectedSolution`.
2. Сначала сделать 2-3 ручных шаблона топологий без случайного графа.
3. Добавить генерацию адресов и LAN-диапазонов.
4. Оставить текущий фиксированный пример как `exam-guide-example`.
5. Добавить кнопку `Новая задача` только после прохождения автотестов генератора.
6. После этого решить, подключать ли v2 к режиму экзамена вместо старой задачи.

## Реализованный генератор v2

После планирования генератор был реализован упрощенно и без полностью случайного графа. Это сознательное решение: для подготовки к экзамену надежнее иметь несколько контролируемых шаблонов, чем сложный генератор с риском спорных маршрутов.

Реализация:

- `routing_task_v2.js` теперь содержит список `ROUTING_V2_TEMPLATES`;
- кнопка `Новая задача` выбирает другой шаблон;
- текущий экзаменационный пример сохранен как шаблон `exam-guide-example`;
- эталонные таблицы по-прежнему строятся алгоритмом через граф, а не ручными ответами;
- порядок строк в таблице для пользователя не важен.

Шаблоны:

- `exam-guide-example` - исходная схема из гайда: `R1-R2`, `R1-R3`, `R2-R3`, `R3-R4`;
- `no-r2-r3-through-r1` - нет линка `R2-R3`, левая ветка идет через `R1`;
- `line-no-r1-r3` - линейная топология `R1-R2-R3-R4`, нет прямого `R1-R3`;
- `three-router-chain` - 3 маршрутизатора цепочкой;
- `star-through-r1` - все ветки идут через `R1`.

Дополнительно сделано:

- схема строится динамически по шаблону;
- для схемы есть горизонтальная прокрутка;
- подписи IP рисуются последним SVG-слоем и имеют обводку, чтобы читаться поверх линий/узлов.
- таблицы маршрутизации теперь стартуют без строк;
- пользователь сам добавляет строки для каждого маршрутизатора;
- пустые случайно добавленные строки игнорируются;
- заполненные лишние/дублирующие строки считаются ошибкой.

## Экзамен v2

Добавлен отдельный режим `Экзамен v2`.

Отличие от обычного тренажера v2:

- после 15 single-choice и 4 multi-choice вопросов открывается новая задача v2;
- в задаче минимум подсказок: условие, схема, поля;
- кнопка `Новая задача` скрыта;
- промежуточные кнопки `Проверить сети`, `Проверить таблицы`, `Проверить оборудование` скрыты;
- остается только финальная кнопка `Завершить экзамен`;
- ошибки и эталонный разбор показываются только на странице результата.

Обычный тренажер v2 при этом остался учебным: там по-прежнему есть `Новая задача` и промежуточные проверки.

Автотест:

- файл: `routing_task_v2.test.js`;
- запуск: `node routing_task_v2.test.js`;
- проверяет все шаблоны без браузера;
- проверяет валидность IP, непересечение сетей, LAN-диапазоны, связность графа, connected/static/default routes и корректность next-hop;
- имитирует пользовательские строки таблиц и проверяет, что эталонный ввод получает полный балл;
- проверяет, что LAN-облака не расположены слишком близко к роутерам;
- последний прогон: `5 templates ok, 106 route rows checked`.
